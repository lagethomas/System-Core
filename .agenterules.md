# ROLE: PHP Senior Developer & System Architect (Thomas's Framework)
# CONTEXT: PHP Core System (MariaDB + Redis) - PHP 8+
# UPDATED: 2026-04-15 - Modular CSS & Component Architecture

## 1. ARQUITETURA E PADRÕES
- **STRICT MODE**: `declare(strict_types=1);` e tipagem obrigatória em todas as funções/métodos.
- **ESTRUTURA**: MVC Modular (Domain-Driven) com PSR-4. Versionamento semântico obrigatório.
- **PERFORMANCE**: Opcache Preloading + Redis para sessões e queries custosas (TTL 3600s).
- **AJAX-FIRST**: 100% das ações via AJAX/Fetch API. Proibido refresh de página para ações de CRUD.
- **DIRETÓRIOS**:
  ├── src/Modules/[Dominio]/ (Controllers, Models, Views)
  ├── public/assets/css/
  │   ├── components/ (Atomic UI: buttons, forms, tables, etc.)
  │   ├── modules/ (CSS específico por módulo)
  │   ├── theme/ (Variáveis de cores/branding)
  │   └── style.css (Apenas layout/esqueleto e variáveis globais)
  └── includes/repositories/ (Abstração de dados global)

## 2. SEGURANÇA E DADOS
- **SQL**: Uso exclusivo de PDO com Prepared Statements. SQL puro com variáveis interpoladas é PROIBIDO.
- **XSS/CSRF**: `htmlspecialchars()` em saídas HTML. Tokens CSRF obrigatórios em TODOS os forms.
- **LOGS GLOBAIS**: Toda criação de log via `LogRepository` deve respeitar a configuração `security_enable_logs`.
- **CRIPTOGRAFIA**: Senhas com `PASSWORD_ARGON2ID`. Dados sensíveis com AES-256.

## 3. CSS MODULAR & COMPONENTES (PADRÃO CORE)
- **CARREGAMENTO**: O `header.php` utiliza um loop automático (`$core_components`) para carregar componentes de `/assets/css/components/`. 
- **NOVOS COMPONENTES**: Devem ser criados como arquivos independentes e registrados no loop do `header.php`.
- **CSS DE MÓDULO**: Estilos que pertencem apenas a uma tela específica devem ficar em `/assets/css/modules/[nome_modulo].css` e serem chamados apenas na View do módulo.
- **COMPONENTES PADRONIZADOS**:
    - **Buttons**: `.btn-primary`, `.btn-secondary`, `.btn-danger`, `.btn-primary-glass` (premium).
    - **Tables**: `.premium-table` dentro de `.table-responsive`. Altura mínima obrigatória de `55vh` para cards com tabela.
    - **Switches**: Estrutura `.switch` + `.slider` (sempre usar o componente `switches.css`).
    - **Uploads**: Padrão `.upload-field` com drag-and-drop e container de preview dinâmico.
    - **Cards**: `.card` (base), `.stat-card-premium` (dashboard com gradientes), `.dashboard-card` (containers de conteúdo).

## 4. INTERFACE (UI/UX PREMIUM)
- **ESTÉTICA**: Design moderno, uso de Glassmorphism, sombras suaves (`0 10px 30px rgba(0,0,0,0.1)`) e bordas arredondadas (`20px`).
- **SELECTABLE CARDS**: Container `.selectable-card` para escolhas visuais (ex: planos, opções de segurança).
- **ESPAÇAMENTO**: Seguir a escala de `gap: 15px/24px/30px`. Padronizar margens nos headers de página (`margin-bottom: 24px`).
- **SCROLLBARS**: Usar o componente `scroll.css` para manter barras de rolagem finas e integradas ao tema.

## 5. PADRONIZAÇÃO DE FORMULÁRIOS
- **FLOATING LABELS**: Obrigatório o uso de `.floating-group`. O input deve ter `placeholder=" "` e vir ANTES da label.
- **AÇÕES FLUTUANTES**: Para múltiplos botões no campo (olho de senha + gerador), usar o container `.floating-actions` para garantir alinhamento lado a lado.
- **GRIDS**: `.form-grid-2` ou `.form-grid-3`. Responsividade automática para 1 coluna em mobile garantida pelo `forms.css`.
- **MÁSCARAS**: Inicializar sempre via `UI.initMasks()` após carregamento dinâmico.

## 6. FEEDBACK (SweetAlert2 & Toasts)
- **SweetAlert2**: Para confirmações e erros críticos (Fundo escuro `#1a1b26`, detalhes dourados `#d4af37`).
- **Toasts**: Para notificações rápidas e feedbacks de salvamento sem interrupção.

## 7. REGRAS DE NEGÓCIO E LOGS
- **AUDITORIA**: Registro obrigatório de logs CUD.
- **NOTIFICAÇÕES**: Gerar registro na tabela `cp_notifications` para cada ação relevante.
- **UPLOADS**: Caminho único: `/public/uploads`. Suporte obrigatório para REMOÇÃO de arquivos (botão "X" no hover).

## 8. MANUTENÇÃO E LIMPEZA (ZERO WASTE)
- **REDUNDÂNCIA**: É PROIBIDO manter estilos no `style.css` que já existem em componentes.
- **LEGADO**: Sempre que refatorar uma funcionalidade, remover código antigo, arquivos de teste e migrations obsoletas.
- **CSS INLINE**: Estritamente proibido. Toda regra deve estar em um componente ou módulo.

## 9. PADRÃO PARA CRIAÇÃO DE NOVOS MÓDULOS (100% READY)
Ao solicitar um novo módulo (ex: financeiro, estoque, etc.), o agente deve entregar a solução completa:
1. **Migration**: Arquivo SQL ou script em `/scripts/migration.php` para criar as tabelas necessárias.
2. **Controller**: Implementado em `src/Modules/[Modulo]/Controllers/` com retorno JSON para AJAX.
3. **Repository**: Criar repositório em `includes/repositories/` para abstrair toda a lógica de banco de dados.
4. **Views**: Implementar `index.php` e modais de CRUD em `src/Modules/[Modulo]/Views/`.
5. **Sidebar**: Adicionar automaticamente o link do novo módulo no arquivo `includes/sidebar.php`.
6. **No-Refresh (AJAX)**: Todo o fluxo (Salvar, Editar, Excluir, Filtrar) deve ser via AJAX. Uso de `UI.initAjaxForm()` ou similar é obrigatório.
7. **Estilos**: Se houver necessidade de CSS específico, criar `public/assets/css/modules/[modulo].css`.

## 10. PADRÕES DE DADOS E VALORES
- **MOEDA**: Utilizar `mask-money` em inputs e formatar via helper PHP no backend.
- **DATAS**: Inputs tipo `date` ou com máscara `.mask-date`. Exibição sempre no padrão local `d/m/Y`.
- **CONSISTÊNCIA**: Todo novo módulo deve herdar a estética "Premium" (cards, sombras, botões padronizados).
